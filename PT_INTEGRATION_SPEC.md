# PT Integration Spec — Demo Databases Feature

This document describes how Prospect Tracker (PT) integrates with Demo Builder's batch demo generation system. PT developers can implement this feature entirely from this spec without consulting Demo Builder source code or asking follow-up questions. All API endpoints are unauthenticated and accept/return JSON. The Demo Builder backend handles all AI extraction, SQL generation, image generation, and tablet deployment — PT only needs to call the endpoints described here and render the UI described below.

---

## Section 1: API Reference

**Base URL**
- Production: `https://demo-builder.vercel.app`
- Development: `http://localhost:3002`

All POST requests require `Content-Type: application/json`. No auth headers required on any endpoint.

---

### POST /api/batch/queue

Enqueue one or more prospects for batch demo generation.

**Request body:**
```json
{
  "prospects": [
    {
      "pt_record_id": "<record.id>",
      "name": "<record.name>",
      "menu_url": "<record.menu_link>",
      "restaurant_type": "<record.type>"
    }
  ],
  "skip_if_exists": true
}
```

- `skip_if_exists: true` (default) — skips records that already have a `done` snapshot. Pass `false` to force re-generation (use for single-row "Retry" and "Generate" actions).
- `menu_url` is required per prospect. Do not include records where `menu_link IS NULL` — filter them out before calling this endpoint.

**Response:**
```json
{ "queued": 3, "skipped": 1, "invalid": 0 }
```

- `queued` — number of new records added to the generation queue
- `skipped` — number of records that already had a ready demo (only nonzero when `skip_if_exists: true`)
- `invalid` — number of records rejected (missing required fields)

---

### GET /api/batch/status

Poll generation status for a list of PT record IDs.

**Request:** `GET /api/batch/status?pt_record_ids=uuid1,uuid2,uuid3`

Pass all visible record IDs as a comma-separated query param. Maximum recommended batch: 200 IDs per request.

**Response:**
```json
{
  "results": [
    {
      "pt_record_id": "uuid",
      "status": "no_snapshot | queued | processing | done | failed",
      "session_id": "uuid or null",
      "error": "string or null",
      "created_at": "ISO string or null"
    }
  ]
}
```

**Status meanings:**

| Status | Meaning |
|--------|---------|
| `no_snapshot` | Never generated — no record exists in Demo Builder for this `pt_record_id` |
| `queued` | In the generation queue; the deploy agent has not started it yet |
| `processing` | Currently being generated by the agent (~30–90s depending on menu format) |
| `done` | SQL and images are ready; can be deployed to the tablet |
| `failed` | Generation failed; `error` field contains a human-readable message |

Records not returned in the `results` array should be treated as `no_snapshot`.

---

### POST /api/batch/load

Trigger tablet deployment for a pre-generated demo.

**Request body:**
```json
{
  "pt_record_id": "uuid",
  "connection_id": "uuid"
}
```

- `connection_id` is optional. If omitted, Demo Builder uses the saved connection named `"Demo Tablet"`. If no such connection exists, returns 400.

**Response — success (200):**
```json
{
  "ok": true,
  "session_id": "uuid",
  "message": "Demo queued for deployment. Ready in ~25 seconds."
}
```

Use `session_id` to poll `GET /api/deploy/status` for progress.

**Response — no snapshot (404):**
```json
{ "error": "No pre-generated demo found for this prospect. Run batch generation first." }
```

**Response — no deploy target (400):**
```json
{ "error": "No deploy target found. Save a connection named 'Demo Tablet' in Demo Builder settings." }
```

---

### GET /api/deploy/status

Poll tablet deployment progress after calling `/api/batch/load`.

**Request:** `GET /api/deploy/status?session_id=<uuid>`

**Response:**
```json
{
  "deploy_status": "idle | queued | executing | done | failed",
  "deploy_result": {
    "ok": true,
    "rows_affected": 0,
    "images_pushed": 0,
    "images_failed": 0,
    "pos_restarted": false,
    "pos_running": false,
    "error": null,
    "timestamp": "ISO string"
  }
}
```

`deploy_result` is null until the deploy agent processes the session. Poll every 3 seconds. Stop polling when `deploy_status` is `done` or `failed`.

---

### GET /api/connections

Fetch saved tablet connections for the deploy target selector.

**Request:** `GET /api/connections`

**Response:** Array of connection objects:
```json
[
  {
    "id": "uuid",
    "name": "Demo Tablet",
    "host": "100.112.68.19",
    "port": 3306,
    "database_name": "PecanDemoDb"
  }
]
```

Pre-select the entry named `"Demo Tablet"` in the connection selector modal.

## Section 2: PT Data Model

The following fields from PT's Supabase schema (`prospect`) are used by this feature.

**`prospect.records`**

| Field | Type | Notes |
|-------|------|-------|
| `id` | UUID | Used as `pt_record_id` in all Demo Builder API calls |
| `name` | string | Restaurant name |
| `menu_link` | string \| null | Menu URL — records without this cannot be batch-generated |
| `type` | string | Free-text restaurant type (Demo Builder normalizes it) |
| `city` | string | Used for area filter in the Demo Databases panel |

**`prospect.canvass_stops`**

| Field | Type | Notes |
|-------|------|-------|
| `id` | UUID | Stop ID |
| `record_id` | UUID | FK → `prospect.records.id` |
| `name` | string | |
| `status` | string | |

**`prospect.leads`**

| Field | Type | Notes |
|-------|------|-------|
| `id` | UUID | Lead ID |
| `record_id` | UUID | FK → `prospect.records.id` |
| `name` | string | |
| `status` | string | `'Open'` \| `'Won'` \| `'Lost'` |

## Section 3: "Demo Databases" Panel (Database Tab)

### Placement

Add a collapsible panel titled **"Demo Databases"** in PT's existing Database tab, below all existing content.

### Panel header

- **Title:** "Demo Databases"
- **Area filter:** Text input that filters the list to records where `city` contains the entered string (case-insensitive, client-side filter)
- **"Generate All" button:** Calls `POST /api/batch/queue` with `skip_if_exists: true` for all visible records where `menu_link IS NOT NULL` and current status is `no_snapshot` or `failed`

### Data loading

On panel mount (and after "Generate All" or any single-row action), fetch all `prospect.records` from PT's Supabase, then immediately call:

```
GET /api/batch/status?pt_record_ids=<all visible IDs joined by comma>
```

Store the status results in local React state keyed by `pt_record_id`. This is the **status cache** referenced throughout this spec.

### Row layout

One row per record. Each row shows:

1. Restaurant name (bold)
2. Truncated `menu_link` — max 40 chars, ellipsis in middle (e.g. `https://menus.restaurant.com/...menu.pdf`). Show `"No menu link"` in gray if null.
3. Status badge
4. Action control

### Row states

Derive each row's display state from two inputs: whether `menu_link` is null, and the `status` from the status cache (default `no_snapshot` if not present in cache).

| Condition | Badge | Action control |
|-----------|-------|----------------|
| `menu_link` is null | Gray pill: "No menu link" | Inline URL input field + "Add & Generate" button (see below) |
| status `no_snapshot` | Gray pill: "Not generated" | "Generate" button |
| status `queued` | Yellow pill: "Queued…" | Spinner, no button |
| status `processing` | Yellow animated pill: "Generating…" | Spinner, no button |
| status `done` | Green pill: "Ready ✓" | "Pre-load" button |
| status `failed` | Red pill: "Failed" with tooltip showing `error` text | "Retry" button |

### Polling

After any generate action (single row or "Generate All"):
- Poll `GET /api/batch/status` for all visible record IDs every **8 seconds**
- Update the status cache with each poll response
- Stop polling when every visible row is `done`, `failed`, or `no_snapshot` (i.e., no rows are `queued` or `processing`)

### "Generate" button (single row)

Call `POST /api/batch/queue`:
```json
{
  "prospects": [{ "pt_record_id": "...", "name": "...", "menu_url": "...", "restaurant_type": "..." }],
  "skip_if_exists": false
}
```
Immediately optimistically set that row's status to `queued` in local state. Start polling.

### "Retry" button (single row)

Same as "Generate" above — identical API call with `skip_if_exists: false`.

### "Generate All" button

Call `POST /api/batch/queue`:
```json
{
  "prospects": [ /* all visible records where menu_link IS NOT NULL */ ],
  "skip_if_exists": true
}
```
Optimistically set all included rows to `queued`. Start polling. Show a toast: `"Queued {queued} restaurant(s) for generation"` using the response counts.

### Inline URL input (rows where menu_link is null)

- Render a small text input inline in the action column
- Show placeholder: "Paste menu URL…"
- Show "Add & Generate" button next to it (disabled until input is non-empty)
- On submit: call `POST /api/batch/queue` with the entered URL as `menu_url`, `skip_if_exists: false`
- **Do not** write the URL back to `prospect.records` — this only triggers generation
- Optimistically set row status to `queued`. Start polling.

## Section 4: "Load Demo" Button on Canvass Stop Cards

### Visibility

Show the "Load Demo" button on a canvass stop card only when **all of the following are true**:
- The stop's `record_id` is present in the status cache
- The cached status for that `record_id` is `done`

Do not show the button if the status is anything else, or if the record is not in the cache.

### Interaction flow

**Step 1 — Connection selector**

On button click, show a small modal or popover:
- Fetch `GET /api/connections` (fetch once and cache for the session)
- List each connection as `{name} — {host}:{port}`
- Default selection: the connection named `"Demo Tablet"`
- Show a "Deploy" confirm button and a "Cancel" button

**Step 2 — Dispatch**

On confirm, call:
```json
POST /api/batch/load
{
  "pt_record_id": "<stop.record_id>",
  "connection_id": "<selectedConnection.id>"
}
```

**Step 3 — Progress display**

Show inline progress on the canvass stop card:

- `ok: true` response received → begin polling `GET /api/deploy/status?session_id=<id>` every **3 seconds**
- `deploy_status: queued` or `executing` → show spinner + text: `"Deploying to tablet… (~25s)"`
- `deploy_status: done` → show `"Live on tablet ✓"` in green; auto-dismiss after 5 seconds
- `deploy_status: failed` → show `"Deploy failed"` in red; show `deploy_result.error` in a tooltip

**Step 4 — Error cases**

Handle these without opening the progress display:
- HTTP 404 from `/api/batch/load` → Show inline error: `"Demo not ready — generate it first"`
- HTTP 400 from `/api/batch/load` → Show the `error` string from the response body verbatim
- Network error → Show: `"Could not reach Demo Builder — check your connection"`

## Section 5: "Build Custom Demo" Button on Lead Cards

### Visibility

Show on **all lead cards**, regardless of snapshot status. There is no condition check needed.

### Behavior

Open Demo Builder in a new browser tab at this URL:

```
https://demo-builder.vercel.app/?menu_url=<encoded>&name=<encoded>&type=<encoded>&pt_record_id=<record_id>
```

Construct the URL as follows:

```js
const params = new URLSearchParams();
if (record.menu_link) params.set("menu_url", record.menu_link);
params.set("name", lead.name);
if (record.type) params.set("type", record.type);
params.set("pt_record_id", record.id);

const url = `https://demo-builder.vercel.app/?${params.toString()}`;
window.open(url, "_blank", "noopener,noreferrer");
```

- If `record.menu_link` is null, omit `menu_url` — Demo Builder will prompt the user to upload a file manually.
- Use `lead.name` (not `record.name`) as the `name` param — leads may have a custom name.
- `pt_record_id` is a UUID and does not need encoding; `URLSearchParams` handles it safely.

### What happens in Demo Builder

Demo Builder reads the URL params on load and pre-fills the extraction step. The user runs through the full AI pipeline (extract → design → deploy). The resulting session is linked to the lead via `pt_record_id`. After the user completes the Demo Builder flow and the generation finishes, the Demo Databases panel in PT will show that record's status as `done` on the next poll.

## Section 6: Important Notes for PT Developers

**1. The deploy agent must be running on the Mac.**
Demo Builder's deploy agent (`agent/deploy_agent.py`) runs as a launchd service on the Mac (`com.valuesystems.demo-builder-agent`). It polls Supabase for queued sessions and executes them. If rows stay stuck in `processing` for more than 5 minutes, the agent is not running. PT cannot start or restart the agent — advise users to check the Mac.

**2. Tablet connectivity requires Tailscale.**
Deployment targets the demo tablet at Tailscale IP `100.112.68.19`. The Mac running the agent must have Tailscale active and the tablet must be online. Advise users to verify Tailscale connectivity before a demo day. Network errors in the deploy step will surface as `deploy_status: failed` with an error message.

**3. Batch-generate the night before, not in front of clients.**
AI extraction takes 30–90 seconds per restaurant. PDFs take longer than URLs. Running "Generate All" for 20+ prospects takes several minutes. The intended workflow is: generate the night before the sales trip, verify all rows show "Ready ✓", then use "Load Demo" in the field for fast (~25s) deployment.

**4. "Load Demo" is destructive — one demo at a time.**
Deploying a demo via "Load Demo" replaces the entire `PecanDemoDb` database on the tablet. The previous demo is gone immediately. There is no undo. Only one demo can be active on the tablet at a time. Communicate this clearly in the UI — the connection selector modal is a good place for a brief warning.

**5. The inline URL input does not persist to `prospect.records`.**
When a user enters a URL in the `no_menu_link` inline input and clicks "Add & Generate", the URL is sent to Demo Builder for extraction only. PT's Supabase record is not updated. If the URL should be saved permanently to the record, that is a separate PT feature not covered by this spec.

**6. Keep the status cache in local state and do not over-fetch.**
Store the `GET /api/batch/status` response in a React state object keyed by `pt_record_id`. Do not call the status endpoint on every render. Fetch on: panel mount, after any queue action, and on the 8-second polling interval. The status cache is also read by canvass stop cards (for "Load Demo" visibility) — share it via context or a shared store rather than fetching separately in each card component.
